# 转播大屏数据对接 (S1 · broadcast/index.html)

> 配套: `static/broadcast/live_adapter.js`(已写好的聚合层) · 字段权威定义 `docs/schema.md` v0.2。
> 目标: 把 index.html 里的内置模拟剧情(SCRIPT)换成真实牌局数据。

## 1. 数据链路与联调方式

```
live_cli --ws ──► ws hub (:8765 收 / :8766 广播) ──► 浏览器 WebSocket
```

Spark 上跑(两个终端或 tmux 两窗格):

```bash
python -m texas_agent.bus                    # 终端1: hub
bash tools/run_live.sh --ws                  # 终端2: 牌局(加 --ws)
```

**前端开发不需要实体牌桌**: 回放器可把任意历史会话按原速/倍速重播进 hub, 消息一模一样:

```bash
python -m texas_agent.bus                    # hub 照常
python tools/replay.py sessions/20260710-223501-drill-live-llm2.jsonl --ws --speed 4
```

浏览器端一行接入(适配层自动重连、聚合成视图模型):

```html
<script src="./live_adapter.js"></script>
<script>
TexasLive.connect({
  url: 'ws://<SPARK_IP>:8766',
  names: ['陈默', '老周', '苏晴', '大飞'],   // 座位0..3 显示名
  onUpdate: vm => this.setState({ live: vm }),
});
</script>
```

## 2. 视图模型 → 大屏元素映射

`onUpdate(vm)` 的 `vm` 字段(全部由适配层从总线消息聚合, 无需自己拼):

| 大屏元素 | vm 字段 | 来源消息 | 备注 |
|---|---|---|---|
| 顶栏 手牌# / 盲注 | `handNo` `blinds` | game_event/hand_start | |
| 街道圆点/名称 | `street` `streetZh` | game_event.state | preflop/flop/turn/river/complete |
| 公共牌 | `board` | game_event.state | `['Jh','8h','3c']` 点数大写+花色小写 |
| 彩池 | `pot` | game_event.state | |
| 座位筹码/下注/在局 | `seats[i].stack/.bet/.inHand` | game_event.state | **座位 0 基** |
| 玩家底牌 | `seats[i].hole` | game_event.state | 上帝模式 `['Ah','Kh']`; 公开模式 `null`; `'??'`=未知 |
| 行动高亮(呼吸金框) | `actor` | input_request | 行动完成后清空 |
| 玩家动作角标 | `acts[seat]` | game_event/action | 已翻译: '加注到 300' |
| 胜率数字/条 | `equity[seat]` | analysis_update | 0~100 整数 |
| **胜率之河** | `history[]` | analysis_update | `{eq:{seat:pct}, mark:'翻牌'\|null}`, 直接喂 buildRiver |
| 解说条(上一句/当前句) | `says[]` | commentary | 取末两条 |
| 机器人状态徽记 | `robot` | 多源 | 待机/发牌中/验证/等待行动/**视觉审计✓⚠**/完成 |
| 荷官提词(可选展示) | `prompt` | dealer_prompt | |
| 赢家高亮/结算 | `winner` `payoffs` | game_event/settlement | winner=payoffs 最大者 |
| 识别来源徽章(演示点!) | `dealSources[zone]` | deal.detail.source | `'vlm'`=VLM 仲裁认牌 `'ops'`=人工补录; 缺省=主路 |
| 视觉审计流水(演示点!) | `audits[]` | audit_report | `{verdict:'match'\|'mismatch', note, board}` |
| 告警 | `alerts[]` | alert / dealer_prompt(level=alert) | |
| 进阶指标(危险牌/坚果/赔率) | `analysis` | analysis_update 原文 | 字段见 schema.md, 公开模式部分为 null → 显示"—" |

**大屏没有数据源的元素**(保留占位或隐藏): VPIP/PFR/AF(需跨手累计, 引擎单手不产)、
心率/Δ心率(面部组另一条 ws, 不经本 hub)、玩家风格标签(前端配置)。

## 3. GTO 数据 (`vm.gto` / `vm.deviations`)

行动前发 `gto_hint`, 行动偏离基线时发 `gto_deviation`(均已聚合进 vm):

```js
vm.gto[2] = {
  node: '9max_100bb/UTG/RFI',          // 位置×行动序列
  hero: '99',                          // 该玩家起手牌格(上帝模式)
  summary: { 'raise_2.5': 0.121, fold: 0.879 },   // 全 range 汇总频率
  matrix: { AA: {'raise_2.5':1.0}, A5s: {'raise_2.5':0.5, fold:0.5}, /*…169 格*/ },
  actions: ['raise_2.5', 'fold'],
  source: 'chart'                      // chart|placeholder|solver_offline(界面须标注)
}
vm.deviations[-1] = { seat:2, hand:'99', gto:{'raise_2.5':1.0}, action:'call' }
```

矩阵渲染约定(schema.md): 13×13, 对角=对子, 右上=同花 s, 左下=非同花 o, hero 格金色描边;
每格按各动作频率横向分色。查不到的节点**不发消息**(界面维持上一状态或收起面板)。
措辞纪律: 一律称 **"GTO 基线/参考策略"**, 不宣称实时求解。

## 4. 注意事项

- **座位号 0 基**(`seat:0`=P1), 展示时 +1 或用 names 映射。
- 断线重连后 `game_event` 自带全量快照, 一条消息即恢复; 但 `history`(胜率之河)
  只从连上时开始积累 —— **开手前先连**。
- `analysis_update` 整包重发, 频率约每次状态变化一次; 适配层已截流至回调级。
- 公共牌翻牌发到一半(1~2 张)时不发 analysis_update(设计如此)。
- 上帝屏(含底牌)只能朝评委不朝牌桌(站位纪律)。
