# Agent 与视觉模块（无机械臂版）· 黑客松 48 小时执行手册

版本 v0.1 · 真人发牌，系统担任全知裁判、导播与解说 · 平行文档：《面部识别模块黑客松执行手册》《牌桌智脑前端设计规范》

---

## 0. 交付物定义（做到什么算完成）

评委看到：真人荷官发牌，系统全程跟踪牌局——语音/屏幕提示荷官下一步该发什么，视觉确认每张牌落位，规则引擎实时推进状态并结算，前端同步显示牌桌、每位玩家的精确胜率（上帝视角）、LLM 解说与行为张力。荷官发错、漏发、识别不确定时系统会发现并引导纠正。整场可回放。

两种运行模式（yaml 一键切换）：

- **上帝模式**（主打）：发牌经过带出口相机的发牌靴，或使用预排牌序——系统知道每一张牌（含底牌），胜率是精确值，这是电视转播级的演示效果；
- **公开模式**（降级）：无靴口信息，只识别桌面上翻开的公共牌，底牌到摊牌才知道，胜率退化为"基于公开信息的估算"。

本文档同时是机械臂版的**热降级方案**：两版共用 hub、消息 schema、规则引擎、分析层与解说层，现场机械臂故障时切到本模式，代码零改动、只换"执行器"。

明确不做：筹码视觉（下注申报制）、GTO 现场实时求解（用预计算查表替代，见 M7）、多桌。

## 1. 架构核心：两个思想

**思想一：人肉执行器。**Agent 的感知→决策→行动闭环完整保留，只是"行动"从驱动机械臂变成**指挥人类**：屏幕大字 + TTS 提词（"请发转牌到 C4"），然后用视觉验证人类照做了。求助、重试、纠错的路由逻辑与机器人版同构——这让 pitch 叙事连贯："同一个 Agent，今天指挥人，明天指挥机械臂"。

**思想二：期望驱动感知。**德州的发牌剧本是完全确定的，引擎永远知道"下一个应该发生的事件"是什么。因此视觉不做开放式检测，只回答一个封闭问题："期望的那件事发生了吗？"——盯住指定区域、等场景静止后核验存在性变化。误报率与开放检测完全不是一个量级，这是无机器人版可靠性的来源。

## 2. 模块分解

### M1 事件 hub 与规则引擎（与机器人版完全共用）

- ws_hub 唯一实例（/pub 8765、/sub 8766），所有消息 seq+ts、逐行落 `sessions/*.jsonl`，回放即演示保险。
- PokerKit NLHE 预设 + 薄适配器：输出"下一个需要的事件"（DEAL/AWAIT/SHOWDOWN）、接收玩家动作、每次变化发 game_event。胜率用 eval7（上帝模式精确枚举，公开模式蒙特卡洛对随机范围）。
- 消息类型沿用前端需求文档 v0.1 §4，外加：

```json
{"type":"agent_trace","ts":...,"text":"提示荷官发转牌；等待 C4 出现"}
{"type":"dealer_prompt","text":"请发转牌 → C4","tts":true}
{"type":"input_request","seat":3,"legal":["fold","call 100","raise"],"timeout_s":30}
```

### M2 顶视感知（Ace Pro 2）

- **标定**：桌垫 6~8 个分散 ArUco → 全点拟合单应（吃掉广角畸变）；`zones.yaml` 定义 C1~C5、P1a/b~P4a/b、MUCK、DECK（荷官手边牌堆区）。竖屏输出记得旋转。
- **稳定门**：全图帧差低于阈值持续 0.8s = 静止。**所有区域核验只在静止帧上做**——荷官的手臂扫过区域时一律不判定，手撤走画面静止才评估。这一条消化了"人在桌上操作"带来的全部遮挡与误报问题。
- **区域核验**：牌背存在 = 牌背模板 NCC > 0.7 或背色 HSV 占比阈值；牌面存在 = 白底占比 + 矩形轮廓。输出三态：无牌 / 背面 / 正面。

### M3 读牌相机（一套匹配器，两个安装位）

匹配器只写一份：固定 ROI → Canny+minAreaRect 细对齐 → 裁角标 → 13 点数模板 NCC + 红黑 HSV 判色 + 4 花色形状 NCC，置信 <0.75 报"不确定"。模板采集工具：交互脚本逐张入库，52 张约 15 分钟。

- **上帝模式 · 靴口位**：微距对准发牌靴出口，荷官每抽一张即被读取（含底牌）。双张粘连 = 侧视两条白边 → 提示荷官重发。
- **公开模式 · 公共牌位**：同一相机（或任意备用 webcam）低架在公共牌槽上方 30~40cm 特写，只读翻开的板面牌。
- **预排牌序**（`god_source: preset`）：赛前按 `deck_order.txt` 理牌，系统按序推断、相机降级为核验。它同时是导演工具——为路演编排一手转牌反超的剧情，胜率之河的戏剧曲线可以被设计。

### M4 荷官提词器

屏幕常驻大字当前指令 + macOS `say` 播报（串行队列）。指令文案与引擎事件一一对应："发底牌：P1 第 1 张""烧牌""发翻牌 → C1 C2 C3""发转牌 → C4"。提词后进入期望窗口（默认 20s），超时轻声重提一次，再超时升级 alert。

### M5 编排器（Agent 本体）

```python
while True:
    need = engine.next_required()          # DEAL(zones) | AWAIT(seat) | SHOWDOWN
    if need.kind == 'DEAL':
        prompt_dealer(need)                              # 屏幕+TTS
        ev = watch(zones=need.zones, expect='appear',
                   timeout=20)                           # 静止帧上核验
        if ev.timeout:      prompt_dealer(need, again=True); continue
        if ev.wrong_zone:   ops.confirm(f"看到牌落在 {ev.zone}，其实是 {need.zone}？")
        card = god.next_card()             # 靴口识别 / 预排 / None(公开模式)
        if card is None and need.face == 'up':
            card = board_cam.read(need.zone)             # 公开模式读板面
        if card == '不确定': card = ops.ask_card()        # 操作员两键输入
        engine.record_deal(card, need.zone)
    elif need.kind == 'AWAIT':
        bus.emit('input_request', seat=need.seat, legal=engine.legal_actions())
        engine.apply(inputs.get(need.seat))              # 键盘主 / 语音弹性
    elif need.kind == 'SHOWDOWN':
        engine.settle()
```

每个分支进出发 agent_trace；纠错走 correction 事件（署名 + 原值 + 新值），审计链与前端设计一致。

### M6 全量分析层与 LLM 解说（与机器人版共用）

原则：**信息集内可算尽算**。上帝模式下唯一的未知只剩公共牌走向（四人局翻牌圈也只有 C(41,2)=820 种），实现要点是**一次枚举、顺路填满**：用 eval7 对全部剩余走向跑一个循环，下表所有指标在同一循环内同时累加，全量计算 <0.5s，每次状态变化整包重发 analysis_update。

**计算纪律**：一切概率只以"牌桌信息集"（各家底牌 + 已发公共牌）为条件，**绝不使用预排牌序的未来信息**——否则胜率变成 0/100 的剧透，分析层的公信力当场清零。

| 指标 | 说明 | 档 |
|---|---|---|
| 精确胜率 win/tie | 每人对全部剩余走向枚举；all-in 后逐张发牌逐次更新（跑马图数据源） | 核心 |
| 终局牌型分布 | 每人到河牌成 高牌~同花顺 各档的概率条（"他 55% 收一对、8% 中三条"） | 核心 |
| 当前牌力与坚果 | 成牌类别、牌力分位、"第 N 坚果"、当前坚果是哪手 | 核心 |
| 领先/落后 + 反超概率 | 上帝模式下领先关系是**确定事实**；落后者标 P(反超)，电视转播的经典图形 | 核心 |
| 干净 / 被污染补牌 | 上帝模式独有：改善的 outs 中剔除"改善但仍输"的张（你成花、他恰好成船） | 核心 |
| 下一张危险牌条 | P(下张成花 / 成顺 / 配对面 / 高张压制)，按真实剩余牌堆精确计数 | 核心 |
| 底池赔率与所需胜率 | 当前行动者，含 SPR | 核心 |
| 剩余牌堆构成 | 各花色/点数还剩几张（"红桃只剩 6 张"），算牌观感、解说素材 | 核心 |
| 牌面质构 | 干/湿评分、听牌密度、坚果易主敏感度（"转牌可能大洗牌"） | 进阶 |
| MDF 与 α | 面对下注的最小防守频率、诈唬盈亏平衡点 = 风险/(风险+回报) | 进阶 |
| 加注 EV 曲线 | 以对手弃牌率为横轴的参数化 EV，前端可做滑杆小玩具 | 进阶 |
| 阻断牌效应 | hero 阻断对手多少坚果组合（公开/复盘模式配范围使用） | 进阶 |
| 运气表 | all-in 时 EV 与实际结果之差逐手累计（"今晚谁被发牌员针对"） | 进阶 |

`analysis_update` 载荷骨架（前端按区取用：行动焦点区吃赔率与补牌，玩家栏吃胜率与牌型分布微条，公共牌区吃危险牌条，摊牌后吃运气表）：

```json
{"type":"analysis_update","street":"flop",
 "board_next":{"flush":0.20,"straight":0.09,"pair_board":0.16,"overcard":0.27},
 "deck_remaining":{"h":6,"s":9,"d":11,"c":9},
 "players":[{"seat":1,"equity":{"win":0.71,"tie":0.01},
   "now":{"cat":"overpair","nut_rank":4,"hs_pct":0.83,"ahead":true},
   "final_dist":{"pair":0.55,"two_pair":0.19,"set":0.08},
   "outs":{"clean":2,"tainted":1},"comeback":null}],
 "actor":{"seat":2,"pot_odds":0.24,"required_eq":0.24,"spr":3.1,"mdf":0.67,"alpha":0.33},
 "nuts":{"desc":"AA 三条","holder":null}}
```

公开模式降级规则：胜率改为对随机/图表范围的蒙特卡洛，领先关系、干净补牌、剩余牌堆三项置空（前端显示"—"），其余照算。

- 解说员：{状态快照, 最近 3 事件, 面部组张力变点} → ≤2 句中文，行为措辞中性，禁编造牌面；**2s 超时或断网切模板句库**（每类事件 3~5 条填空式），TTS 用 `say`。解说的素材质量随上表水涨船高——"红桃只剩六张，P2 的花听只有干净的五张补牌"这种句子，模板和 LLM 都能直接从载荷拼出来。

### M7 GTO 基线层（预计算查表，零现场求解）

对标范围查看器（GTO Wizard 类界面）的效果，黑客松的正确实现是**查表，不是求解**——翻前 GTO 解本来就是静态数据，现场只需要数据、接入、渲染三件事。

- **数据**：`charts/{node}.json`，node 键 = `9max_100bb/{位置}/{面对序列}`，位置沿用 枪口/枪口+1/中位/低劫持/高劫持/关煞/庄位/小盲/大盲。每张表 169 个标准起手牌 → 动作频率（混合策略天然支持）：

```json
{"node":"9max_100bb/枪口/RFI",
 "actions":["raise_2.5","fold"],
 "range":{"AA":{"raise_2.5":1.0},"A5s":{"raise_2.5":0.55,"fold":0.45},"Q8s":{"fold":1.0}}}
```

覆盖集（黑客松够用）：9 个位置的 RFI，加上庄位与大盲面对单次开局加注的 3bet/跟注/弃牌，共 12~15 张表。数据来源两条合法路径：把公开发表的免费图表手工转录成 JSON（半天，可两人分摊），或核对许可后采用开源求解器项目随附的翻前图表；**不要抓取商业产品的解**。

- **运行时接入**（编排器 AWAIT 分支，翻前）：由（座位位置, 行动序列）拼 node 键 → 查表 → 发 `gto_hint`（整张矩阵 + hero 手牌格 + 汇总频率）；玩家行动后发 `gto_deviation`（基线频率 vs 实际动作），同时喂解说员（"GTO 在这个位置只有 10% 的加注频率，P3 选择了加注"）与复盘。查不到的行动序列（罕见 limp 链等）就静默不显示，**绝不编造**。

```json
{"type":"gto_hint","seat":3,"node":"9max_100bb/枪口/RFI","hero":"Q8s",
 "summary":{"raise_2.5":0.101,"fold":0.899},"matrix":{"...":{}},"source":"chart"}
{"type":"gto_deviation","seat":3,"hand":"Q8s","gto":{"raise_2.5":0.0,"fold":1.0},"action":"raise"}
```

- **翻后**：不求解。主显精确胜率（上帝模式）、底池赔率、outs。唯一例外是预排剧情牌——其关键节点已在 D0 用 TexasSolver 离线求解并缓存，现场按 node 命中即显示，界面标注"离线求解"。这是黑客松的魔术：**预排牌序 + 赛前离线求解 = 现场看起来像实时 GTO，而算力为零**。
- **前端组件规格（交第三部分实现）**：13×13 矩阵——对角线为对子、右上三角为同花 s、左下为非同花 o；每格按动作频率**水平比例分割填色**（混合策略一目了然）；hero 手牌格金色描边高亮；矩阵顶部一条汇总频率条（如"加注 10.1% / 弃牌 89.9%"）；hover 显示该格频率与（若有）EV。动作配色定义为语义 token，交设计侧随 tokens 定稿。
- **措辞纪律**：界面与 pitch 一律称"GTO 基线 / 参考策略"，剧情牌节点标注"离线求解"，不宣称现场实时求解——评委里若有懂行的，这一条保住全场可信度。

| 识别"不确定" | 暂停推进 + 操作员两键输入 | <10s | |
| 连抽两张（上帝模式） | 白边检测 → 提示重发 | 荷官收回一张 | <15s |
| 顶视相机拔线 | 前端 stale 标记 | 切全系统 jsonl 回放 | <10s |
| LLM 断网 | 模板解说顶上 | 无 | 0 |

用预排剧情牌录制黄金回放；完整彩排 ×2（一次注入发错槽）。

## 6. 砍单顺序（从上往下砍）

1. 语音申报动作（键盘桩为主路）
2. 弃牌视觉检测（本就申报制，此条只是确保没人顺手去做）
3. 剧情牌离线求解展示（翻前查表基线保留）
4. LLM 解说（模板全程顶替）
5. GTO 翻前基线（矩阵与偏差事件；精确胜率与赔率保留）
6. 读牌相机识别 → 切 preset 预排模式（识别改口头核验）
7. 上帝模式 → 公开模式（只读板面，胜率降级为估算）
8. 四人局 → 两人单挑

**不可砍核心**：期望驱动的发牌跟踪 + 提词纠错闭环 + 精确胜率上屏 + 全系统回放。这四样构成"全知 AI 裁判"故事的骨架。

## 7. 与机器人版的切换关系

编排器中只有 DEAL 分支的"执行器"不同：机器人版调 skills.exec 后视觉**验证**；本版 prompt_dealer 后视觉**检测**。两者共用 watch/verify 的同一套区域核验代码。建议实现为 `executor: robot | human` 的配置开关——现场机械臂任何环节掉链子，改一行配置即刻切人肉模式续演，这本身就是可以写进 pitch 的容错设计。
